AI Gateway Patterns

Which gateway belongs in your architecture — and where?

Four patterns, four different governance layers. Understanding when to apply each one.

↓ Explore the patterns

Four patterns, four different governance layers

Every AI request falls into one of four categories. The wrong choice adds latency, breaks auth, or creates a governance blind spot.

Watch the visual on the right as each pattern appears. Each row represents a different traffic flow and governance model.

Key insight: Not all traffic needs a gateway. Understanding which pattern applies to your use case is the first step to proper architecture.

Pattern 1: Internal Traffic — Why No Gateway

The OBO token chain: the calling user's OAuth token flows end-to-end. Unity Catalog enforces row filters and column masks as that user at query time.

A gateway in this path has two bad options:

  • Pass through: Adds latency, provides zero governance value
  • Re-issue its own token: User identity is lost; current_user() evaluates to gateway identity, breaking row filters silently
UC enforcement happens at the data plane. A gateway can observe HTTP traffic. It cannot see which rows were filtered.

Direct OBO: Governance Enforced Correctly

Without the gateway: the user token arrives at Genie unchanged. current_user() returns reviewer@company.com. The row filter returns only that reviewer's cases.

This pattern works for:

  • Genie and FM APIs
  • Serving endpoints
  • Custom MCP servers on Databricks Apps
  • Any internal Databricks service call
Databricks Apps already has a platform-managed proxy that handles OAuth validation and identity injection. An external gateway creates a redundant, conflicting auth layer.

Pattern 2: LLM Endpoint Governance

Databricks AI Gateway is configured directly on serving endpoints. It governs LLM traffic at the point of consumption — before the model processes the request.

Unity Catalog aware: Rate limit policies reference Databricks users and groups — the same identity model as row filters and column masks.

Four key capabilities:

  • Rate Limiting — Control consumption per user / group / endpoint
  • Guardrails — Input + output content filtering
  • Usage Tracking — Token usage flows to system tables + MLflow
  • Fallback Routing — Primary → backup on error/latency

Pattern 3: Outbound External — UC Connections Proxy

Application code never calls external services directly. It calls a Databricks-managed proxy endpoint, which checks USE CONNECTION authorization and injects credentials before forwarding.

The credential is stored encrypted in Unity Catalog. Application code never receives the raw value.

Instant access control: REVOKE USE CONNECTION = immediate access removal. No redeploy, no code change required.

Two layers, two threat models

Serverless Network Policies (SNP): Workspace-level FQDN allowlist. Defines the approved destination universe. Any destination not on the list is unreachable at the network layer.

UC Connections: Per-app credential authorization. Which app SP can authenticate to which specific service — enforced before any network traffic.

The objection answered: "Your network controls are workspace-level. We need per-app egress controls." Answer: SNP handles the destination universe; UC Connections handles per-app credential authorization.

Governance assumption: Credentials must live exclusively in UC Connections (not env vars or secrets) — enforced by CI/CD secret scanning + system.access.audit.

Pattern 4: Inbound External — External API Gateway

This is where an external gateway genuinely belongs: when requests originate outside Databricks and need a managed facade.

Provides:

  • Auth translation (API key / enterprise SSO → Databricks token)
  • Per-tenant rate limiting
  • API versioning
  • Developer portal

Important: Unity Catalog still governs what the endpoint returns. The gateway handles the boundary; UC handles the data.

Patterns 2 and 4 are additive: External gateway + Databricks AI Gateway can both be in the path — each governs its own layer.

Databricks AI Gateway vs External API Gateway

They're complementary, not competing. Each governs its own layer.

  • Databricks AI Gateway: Inside-out — governs LLM consumption from within, UC-aware
  • External API Gateway: Outside-in — manages external client access, identity-agnostic for Databricks internals
