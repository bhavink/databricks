The Problem

AI applications combine services that were never designed to share a single identity model

Multiple Services, Multiple Identities

  • Genie uses on-behalf-of (OBO) tokens
  • Vector Search uses machine-to-machine (M2M)
  • Custom MCP Servers use proxy identity
  • External APIs use connections

The Challenges

  • Audit trail: Who accessed what data, and when?
  • Confused deputy: A service principal with broad permissions acting on an unauthenticated request
  • Least privilege: How do you scope access when each service has different mechanisms?
Each service was built independently. Your AI application stitches them together. Authorization must be intentional, not accidental.

Three Identity Models

Every Databricks AI application uses one or more of these patterns

Model Who Executes UC Sees Audit As Use When
True OBO User's token Human identity Human email User-specific data access, row/column security
Proxy + M2M Service principal SP identity + proxy header SP (email in app log) Most Databricks Apps today
Pure M2M Service principal SP identity only SP System operations, no user context

Key insight: Only True OBO preserves the full chain of custody from UI click to data access. The others require application-level logging to close the audit gap.

What the Proxy Does

Authenticates

Validates the user's session against Databricks IAM

Strips & Injects

Removes original auth headers. Injects X-Forwarded-Email and X-Forwarded-User

Issues Token

Your app receives the SP's M2M token — or the user's OBO token if User Authorization is configured

# Headers your app.py receives:
X-Forwarded-Email: jane.doe@company.com
X-Forwarded-User: jane.doe
X-Forwarded-Access-Token: eyJ...  # SP token (or OBO if authorized)

How Identity Flows

  1. Proxy 1 authenticates user, injects X-Forwarded-Email
  2. Streamlit reads user identity, forwards token to MCP server
  3. MCP server receives X-Forwarded-Email from Proxy 2
  4. Identity preserved across both hops

Critical Configuration

# MCP Server app.yaml
command:
  - "uvicorn"
  - "server:app"
authorization: "disabled"
# ↑ Allows the forwarded token
#   to pass through Proxy 2

Note: Setting authorization: disabled means Proxy 2 does not issue its own token — the Streamlit app's forwarded token is preserved.

What UC Adds

  • Checks USE CONNECTION permission before forwarding
  • Governance at the platform layer, not the app layer
  • GRANT / REVOKE instantly controls access
  • Connection owner has implicit USE CONNECTION

The Key Principle

The MCP server itself needs zero data permissions for proxied calls. UC validates the caller's right to connect — the server's tools handle execution with their own credentials.

This is how you prevent the confused deputy: the platform gates access before the request ever reaches your code.

OBO vs M2M Decision Matrix

Choose the right identity model for each operation

OBO
User sees their own data — row-level security, column masking, personal dashboards
Requires User Authorization with sql scope. current_user() returns human email.
M2M
Everyone sees the same data — shared dashboards, aggregated metrics, system catalogs
Service principal executes. Supplement with application-level audit logging.
OBO
Need SQL as the user — queries must fire row filters and column masks as the human identity
Requires UI-based User Authorization (not CLI) to get the sql scope in the JWT.
M2M
System-level operations — workspace administration, job scheduling, infrastructure management
No user context needed. SP credentials are appropriate.
Hybrid
Mixed workloads — OBO for data queries, M2M for system calls, fallback when OBO scope is missing
Separate your MCP tools into OBO and M2M categories. Fall back gracefully.

User Authorization

The mechanism that produces real OBO tokens for Databricks Apps

UI vs CLI — A Critical Distinction

UICLI
user_authorized_scopes ✓ ✓
effective_user_api_scopes ✓ ✗
Real OBO JWT ✓ ✗
OBO SQL works ✓ ✗

Takeaway: Always use the UI to configure User Authorization. CLI sets the field but does not produce effective scopes.

What Happens with OBO SQL

-- With OBO + sql scope:
SELECT current_user()
-- → jane.doe@company.com

-- Row filters fire as Jane
-- Column masks apply to Jane
-- Audit log records Jane

Available Scopes

  • sql — Execute SQL as user
  • dashboards.genie — Genie conversations
  • serving.serving-endpoints — Model serving
  • unity-catalog — UC API access

OAuth Scopes Reference

Scopes control what actions an OBO token can perform

Scope Enables Required For
sql Execute SQL statements as the authenticated user OBO queries, row filters, column masks, current_user()
dashboards.genie Interact with Genie conversations Genie API calls on behalf of user
serving.serving-endpoints Query model serving endpoints Calling foundation models or custom endpoints as user
catalog.connections Manage UC connections Creating or modifying connections via user token
unity-catalog Access Unity Catalog APIs Browsing catalogs, schemas, tables as user

Principle of least privilege: Request only the scopes your application actually needs. Each scope is an additional surface area for the token.

Custom MCP Server Patterns

Organizing tools by identity model within a single server

OBO Tools

  • Use the forwarded user token for data queries
  • Read X-Forwarded-Email for identity
  • current_user() returns human email
  • Row filters and column masks fire as user
# OBO tool pattern
token = request.headers[
  "X-Forwarded-Access-Token"
]
result = execute_sql(token, query)

M2M Tools

  • Use SP credentials for system operations
  • Log caller identity from proxy headers
  • Appropriate for workspace-level operations
  • Supplement with application audit logging
# M2M tool pattern
sp_client = WorkspaceClient()
caller = request.headers[
  "X-Forwarded-Email"
]
log_access(caller, action)

Hybrid Pattern

Try OBO first. If the token lacks the required scope, fall back to M2M with enhanced audit logging. Use ContextVar propagation in pure ASGI middleware to thread identity through async handlers.

Audit Trail Architecture

Two layers of audit — platform and application — with a gap between them

Platform Audit

system.access.audit

  • Automatic for all OBO operations
  • Records human email, action, timestamp
  • UC-level granularity
  • No code changes needed
OBO Native

Application Audit

Custom logging in your app

  • Required for M2M paths
  • Log X-Forwarded-Email + action
  • Write to a Delta table or external system
  • Correlate with MLflow trace IDs
M2M Required

The Gap

Platform + Application join

  • No native join key between system.access.audit and MLflow traces
  • Workaround: embed trace_id in SQL comments
  • OBO SQL solves M2M audit gap directly
Manual Correlation

Strategy: Use OBO SQL where possible to get automatic platform audit. For M2M paths, build application-level logging and embed correlation IDs for manual joins.

Best Practices

A checklist for building well-governed AI applications

Identity & Access

  • Use OBO where possible — it preserves human identity through the full stack
  • Minimize SP grants — principle of least privilege for service principals
  • Use UI-based User Authorization for configuring OAuth scopes
  • Validate X-Forwarded-Email in every request handler
  • Separate OBO tools from M2M tools in your MCP server

Governance & Audit

  • Use USE CONNECTION for external service governance
  • Build application-level audit logging for M2M paths
  • Embed trace IDs in SQL comments for correlation
  • Use UC as the governance layer — not application code
  • Review system.access.audit regularly for anomalies

Known Gaps & Workarounds

Honest assessment of current limitations and practical solutions

# Gap Status Workaround
1 M2M audit trail — SP operations don't record human identity in platform audit Solvable Use OBO SQL with sql scope; current_user() returns human email, platform audit records it
2 MLflow-UC join — No native correlation between MLflow traces and system.access.audit Open Embed trace_id in SQL comments; manual join on timestamp + identity
3 sql scope via CLI — CLI sets user_authorized_scopes but doesn't produce effective scopes Resolved Use UI-based User Authorization; CLI behavior is documented
4 is_member() in Genie OBO — Group membership function behaves differently in Genie context Genie-specific Use explicit WHERE clauses with current_user() instead of is_member()
5 authorization:disabled dual use — Can't serve both proxied and direct traffic on same app Open Deploy two app instances, or implement fallback logic in middleware
